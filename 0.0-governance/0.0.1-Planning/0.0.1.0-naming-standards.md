## Overview

This document establishes the naming conventions and formatting standards for all documentation within the Hana-X Governance repository. Consistent naming and formatting ensure:

- **Discoverability**: Easy to locate documents through logical hierarchical organization
- **Maintainability**: Clear structure simplifies updates and versioning
- **Professional Quality**: Consistent presentation across all documentation
- **Collaboration**: Team members can quickly understand document relationships
- **Automation**: Predictable patterns enable tooling and automation

### Core Principles

1. **Hierarchical Numbering**: Use nested numbering to reflect document relationships
2. **Descriptive Names**: Names should clearly indicate content and purpose
3. **Consistency**: Follow established patterns throughout the repository
4. **Simplicity**: Avoid redundant prefixes and suffixes
5. **Readability**: Use kebab-case for multi-word names

## Directory Naming Conventions

### Top-Level Governance Structure

**Pattern**: `0.0.#-[PhaseName]/`

```
0.0-governance/
├── 0.0.1-Planning/
├── 0.0.2-Architecture/
├── 0.0.3-Development/
├── 0.0.4-Testing/
├── 0.0.5-Delivery/
└── 0.0.6-Templates/
```

**Rules**:
- **Numbering**: `0.0.#` where # is sequential (1-6 for SDLC phases)
- **Separator**: Single hyphen `-` between number and name
- **Case**: PascalCase for phase names
- **Order**: Reflects Software Development Lifecycle sequence

### Sub-Directory Structure

**Pattern**: `0.#-[category-name]/`

```
0.0.5-Delivery/
├── 0.1-agents/
├── 0.2-credentials/
├── 0.3-infrastructure/
├── 0.4-service-operations/
└── 0.5-integrations/
```

**Rules**:
- **Numbering**: `0.#` for first-level subdirectories
- **Separator**: Single hyphen `-`
- **Case**: kebab-case for category names (all lowercase, hyphens between words)
- **Order**: Logical grouping, sequential numbering

### Deep Hierarchy

**Pattern**: `0.#.#-[subcategory-name]/`

For nested subdirectories requiring additional organization:

```
0.0.2-Architecture/
└── 0.7-architecture-knowledge/
    ├── 0.7.1-design-patterns/
    └── 0.7.2-technical-decisions/
```

## File Naming Conventions

### Standard Documents

**Pattern**: `0.0.#.#-[descriptive-name].md`

```
0.0.1-Planning/
├── 0.0.1.0-naming-standards.md
├── 0.0.1.1-project-scope.md
├── 0.0.1.2-deployment-methodology.md
├── 0.0.1.3-work-methodology.md
└── 0.0.1.4-traceability-matrix.md
```

**Rules**:
- **Numbering**: `0.0.#.#` where first # matches parent directory, second # is sequential
- **Special**: Use `.0` for meta-documents (standards, guides, README files)
- **Separator**: Single hyphen `-` between number and name
- **Case**: kebab-case for document names (all lowercase, hyphens between words)
- **Extension**: Always `.md` for Markdown files
- **Length**: Keep names concise but descriptive (prefer 2-4 words)

### Templates

**Pattern**: `t-0.##-[template-name]-template.md`

```
0.0.6-Templates/
├── t-0.4-defect-log-template.md
├── t-0.5-task-tracker-template.md
├── t-0.11-summary-task-template.md
└── t-0.13-individual-task-template.md
```

**Rules**:
- **Prefix**: `t-` indicates template file
- **Numbering**: `0.##` sequential across all templates (can use double digits)
- **Suffix**: `-template` at end of name
- **Purpose**: Reusable document frameworks

### Agent Documentation

**Pattern**: `agent-[name].md` or `hx-agent-[topic].md`

```
0.1-agents/
├── agent-alex.md
├── agent-zero.md
└── hx-agent-constitution.md
```

**Rules**:
- **Prefix**: `agent-` for individual agent profiles
- **Prefix**: `hx-agent-` for cross-agent governance documents
- **Case**: kebab-case for names
- **Consistency**: All agent files use same pattern

### Special Cases

**Configuration/Setup Files**:
```
README.md                    # Directory overview (always capitalized)
.gitignore                   # Standard config files (as needed)
```

## Document Structure Standards

### Required Header Section

Every governance document MUST begin with a metadata header:

```markdown
# Document Title

**Document Type**: [Type]  
**Created**: [Date]  
**Last Updated**: [Date]  
**Topic**: [Topic]  
**Purpose**: [Brief purpose statement]  
**Classification**: [Classification Level]  

---
```

**Header Fields**:
- **Document Type**: Plan | Specification | Template | Standards | Reference | Guide
- **Created**: YYYY-MM-DD format or Month DD, YYYY
- **Last Updated**: Optional, use for living documents
- **Topic**: High-level subject area
- **Purpose**: 1-2 sentence description of document purpose
- **Classification**: Internal - Governance | Internal - Operations | Public

### Table of Contents

For documents with 5+ major sections, include a TOC:

```markdown
## Table of Contents

1. [Section Name](#section-name)
2. [Another Section](#another-section)
   - [Subsection](#subsection)
```

### Document Body Structure

```markdown
## Overview
[High-level introduction]

## [Major Section 1]
[Content]

### [Subsection 1.1]
[Content]

## [Major Section 2]
[Content]

---

## Appendices
[Optional supporting content]

---

**Version**: 1.0  
**Maintained By**: Hana-X Governance Team  
**Classification**: Internal - Governance
```

## Markdown Formatting Guidelines

### Headings

- **H1 (`#`)**: Document title only (once per document)
- **H2 (`##`)**: Major sections
- **H3 (`###`)**: Subsections
- **H4 (`####`)**: Sub-subsections (use sparingly)
- **Never skip levels**: Don't jump from H2 to H4

### Text Formatting

- **Bold**: `**text**` for emphasis, labels, important terms
- **Italic**: `*text*` for subtle emphasis (use sparingly)
- **Code**: `` `text` `` for filenames, commands, variables, symbols
- **Strikethrough**: `~~text~~` for deprecated content

### Lists

**Unordered Lists**:
```markdown
- First item
- Second item
  - Nested item
  - Another nested item
- Third item
```

**Ordered Lists**:
```markdown
1. First step
2. Second step
   1. Sub-step
   2. Another sub-step
3. Third step
```

### Links

**Internal Links**:
```markdown
[Link text](#section-anchor)
[Other document](./0.0.1.2-deployment-methodology.md)
```

**External Links**:
```markdown
[Website name](https://example.com)
```

### Tables

```markdown
| Column 1 | Column 2 | Column 3 |
|----------|----------|----------|
| Data 1   | Data 2   | Data 3   |
| Data 4   | Data 5   | Data 6   |
```

**Rules**:
- Use tables for structured data and comparisons
- Include header row
- Align separator row with pipes
- Keep content concise

### Blockquotes

```markdown
> Important note or quoted content
> Can span multiple lines
```

Use for:
- Important warnings or notes
- Quoted specifications
- Highlighted requirements

### Horizontal Rules

```markdown
---
```

Use to:
- Separate header from body
- Separate body from footer
- Divide major document sections

## Code Block Standards

### Language-Tagged Code Blocks

Always specify the language for syntax highlighting:

````markdown
```python
def example_function():
    return "Hello, World!"
```
````

````markdown
```bash
#!/bin/bash
echo "Example script"
```
````

````markdown
```yaml
key: value
nested:
  - item1
  - item2
```
````

### Supported Languages

Common language tags used in Hana-X documentation:

- `bash` / `sh` - Shell scripts and commands
- `python` - Python code
- `yaml` - YAML configuration files
- `json` - JSON data
- `javascript` / `typescript` - JS/TS code
- `sql` - Database queries
- `markdown` - Markdown examples
- `plaintext` - Unformatted text

### Inline Code

Use backticks for inline code references:

```markdown
The `configure_server.sh` script should be run with `sudo` privileges.
Use the `--verbose` flag for detailed output.
```

### File Paths

Always use inline code for file paths:

```markdown
Located in `/srv/cc/Governance/0.0-governance/0.0.1-Planning/`
Edit the file `0.0.1.3-work-methodology.md`
```

## Metadata Requirements

### Document Footer

Every governance document SHOULD end with a footer:

```markdown
---

**Version**: [Version Number]  
**Maintained By**: [Team/Individual]  
**Related Documents**: [Links to related docs]  
**Classification**: [Classification Level]  
**Last Review**: [Date]
```

### Version Numbering

Use semantic versioning for governance documents:

- **1.0**: Initial release
- **1.1**: Minor updates (clarifications, small additions)
- **2.0**: Major revisions (structural changes, significant content updates)

### Related Documents

Link to related governance documents:

```markdown
**Related Documents**:
- [Project Scope](./0.0.1.1-project-scope.md)
- [Work Methodology](./0.0.1.3-work-methodology.md)
- [Test Plan Template](../0.0.6-Templates/t-0.12-test-plan-template.md)
```

## Validation Checklist

Use this checklist when creating or updating governance documents:

### Naming
- [ ] File uses correct hierarchical numbering pattern
- [ ] File name uses kebab-case (all lowercase, hyphens)
- [ ] No redundant prefixes (hx-, hana_x) in filename
- [ ] Template files use `t-` prefix and `-template` suffix
- [ ] File extension is `.md`

### Structure
- [ ] Document has required metadata header
- [ ] H1 title matches document purpose
- [ ] Table of contents included (if 5+ sections)
- [ ] Logical section hierarchy (H2 → H3 → H4)
- [ ] Horizontal rules separate header/body/footer
- [ ] Footer includes version and metadata

### Formatting
- [ ] Code blocks use language tags
- [ ] File paths use inline code formatting
- [ ] Tables properly formatted with headers
- [ ] Lists properly indented
- [ ] Links use descriptive text (not "click here")
- [ ] No trailing whitespace

### Content
- [ ] Purpose clearly stated
- [ ] Target audience identified (if applicable)
- [ ] Examples provided where helpful
- [ ] Related documents linked
- [ ] Version number included

### Quality
- [ ] Spelling and grammar checked
- [ ] Consistent terminology throughout
- [ ] Professional tone maintained
- [ ] No placeholder content in final version
- [ ] Document tested/reviewed before publication
